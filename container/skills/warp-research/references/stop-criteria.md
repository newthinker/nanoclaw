# 停止判据 — 何时调研够了

调研最大的失败模式是**无限搜索**。本文件给出"已足够"的判据和**硬上限**。

## "够判断就停"的信号

- 再搜只是重复已知信息(边际收益趋零)。
- 已能给出**带明确立场**的 TL;DR。
- 关键结论已满足下表对应深度档的交叉验证要求。

## 三档深度的"已足够"判据

| 深度 | 已足够 | 交叉验证要求 |
|------|--------|--------------|
| **overview** | 每个核心子问题各有 1 个可信来源已答;3–4 题覆盖 | 不强制 |
| **standard**(默认) | 每个 priority≥中 的子问题有答案;5–7 题覆盖;矛盾已标注 | 关键结论 2+ 独立源 |
| **deep** | 全部子问题已答;正反观点均覆盖;局限性可明确写出 | 关键结论一手来源交叉验证 |

## ⚑ Token 预算(容器内,最关键的硬约束)

**实测教训**:容器内单次会话 token 有限。一次 standard 档深度 web 调研抓取过多,累计约
**136K tokens → 触发 context 压缩 → 最终没能完成归档**(整篇调研白做)。归档没落地 = 任务失败。
所以**控制 token 比"调研得全"更重要**:

- **vault 优先,web 克制**:vault 复用最省 token(已有笔记直接读)。先把 vault 已有知识吃透,
  web 只补真正缺的那几点,不要拿 web 当主力来源海搜。
- **web 来源按"篇"硬限**:overview **≤ 4 篇** / standard **≤ 8 篇** / deep **≤ 12 篇** 独立来源
  (整篇全文抓取按 token 计极贵;命中要点即停,别整页反复读)。
- **优先选低 token 档**:能 overview 落地,就不要 standard 跑飞。重大决策才上 deep,且 deep 更要省着抓。
- **预算告急先归档**:感觉上下文在变长/接近压缩,**立即停止采集,带现有结论进入成稿+归档**,
  把没查到的标 `info-gap`。**绝不为完整性牺牲归档**。

## 防无限调研硬上限(写死,务必遵守)

- 单子问题搜索轮次 **≤ 3**;单题 web 抓取 **≤ 3 篇**(原 6 偏多,收紧)。
- 子问题树总数 **≤ 10**(动态新增不得突破)。
- 触发范围蔓延(题数 > 10 或来源超上表 2×)→ 回到 `motivation` 校准,砍掉无关分支;
  **宁可对缺口标 `info-gap` 也不无限搜**。

## 降级出口

任一硬上限触顶 **或 token 预算告急** → 停止采集,带着已有信息进入分析/成稿/**归档**,
在报告"局限性"里写明哪些子问题是 `info-gap`。归档(`selvage_call`)永远是流程的必达终点。
